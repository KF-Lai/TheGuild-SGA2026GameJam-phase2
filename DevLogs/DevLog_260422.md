## 規劃：FT-07~FT-11 設計先於實作

### 決策

**先完成 FT-07~FT-11 五份 GDD 設計，再進入 F-01 實作。**

### 理由

1. **介面穩定性**：F-03 已證明「Foundation 層被未實作的下游 API 反噬」的風險——若不先設計 FT-07（建築）、FT-08（職員）、FT-09（陣營）、FT-10（Save/Load）、FT-11（離線結算器），F-03/F-05 等已設計系統可能需要回頭改 API 形狀。
2. **Save/Load 跨系統約束**：FT-10 會要求所有系統正規化存檔 schema；等 FT-10 設計完成再動實作，避免每個系統各自想像序列化策略後大返工。
3. **金流完整性**：FT-07/FT-08 是 FT-05 §6.4 反向依賴清單的兩個主要未設計項，沒有它們，FT-05 的維護費/薪水分支永遠只能跑 stub。
4. **FT-11 離線處理**：Jam 階段雖列為範疇外，但 FT-03 NPC 自主接單的離線邏輯仍需 FT-11 骨架（至少介面契約）才能不留坑。

### 設計順序（建議）

| 順序 | 系統 | 設計重點 |
|---|---|---|
| 1 | FT-10 Save/Load | 序列化 schema 最先定，後續 GDD 以此對齊 |
| 2 | FT-07 Guild Building | 建築項目、升級效果、維護費發布時機 |
| 3 | FT-08 Guild Staff | 職員招募、薪水發布時機、加成查詢介面 |
| 4 | FT-09 Faction Story | styleTag 累積、路線切換、陣營分數推送至 C-06 |
| 5 | FT-11 Offline Resolver | 骨架定義（介面契約 + Jam 外實作排期）|
