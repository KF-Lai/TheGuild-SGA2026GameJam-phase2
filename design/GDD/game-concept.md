# 遊戲概念：The Guild

_建立時間：2026-04-19_
_狀態：草稿_

---

## 概念

> 一款放置型公會經營遊戲，你身為公會長，透過審核委託、建設公會、招募職員來間接影響冒險者的命運——但你無法直接指揮他們。每次打開遊戲，都可能是好消息，也可能是永別。它活在你的桌面上，像一扇通往冒險世界的窗口。

## Core Fantasy

你是一位新上任的公會長。你的公會還很小、很破舊，但冒險者們帶著各自的故事來到這裡。你無法控制他們——他們有自己的個性、自己的判斷——但你可以打造一個讓他們願意留下、願意冒險、願意信任你的公會。

每一個你審核通過的委託、每一座你建造的設施、每一位你招募的職員，都在無形中塑造這個公會的命運，以及每一位冒險者的生死。

你不是英雄，你是讓英雄誕生（或殞落）的人。

---

## 遊戲的獨特性

像 Majesty 一樣你無法直接控制冒險者，而且它活在你的桌面上，讓你在工作間隙偷看你的冒險者是否平安歸來——而你的每一個經營決策，都在默默推動著公會走向不同的陣營命運。

## 玩家體驗分析（MDA Framework）

### 目標美學（玩家的感受）

| 美學面向 | 優先度 | 實現方式 |
| -------------------------------------------- | -------- | ---------------------------------------------- |
| **感官愉悅** | 5 | 日系動漫風角色、桌面透明視窗的視覺新鮮感 |
| **角色扮演** | 2 | 身為公會長，看著冒險者在你的桌面上來去、生死 |
| **敘事劇情** | 1 | 陣營劇本路線因經營決策而分支，冒險者的個人故事 |
| **挑戰與精通** | 4 | 資源管理的壓力、風險評估的決策 |
| **情感連結** | 6 | 與 NPC 冒險者建立情感連結（非社交功能） |
| **探索與發現** | 3 | 新冒險者的故事、新委託帶來的陣營劇情發展 |
| **自我表現** | 7 | 公會建設方向反映玩家風格 |
| **放鬆與舒適** | 8 | 掛機本身的放鬆感（但有後果的緊張打破純放鬆） |

### 關鍵動態（新興玩家行為）

- 玩家會開始「偏愛」某些冒險者，不捨得派他們去危險任務
- 玩家會在派遣前仔細評估任務風險，嘗試用公會建設降低風險
- 玩家會在工作中想著「我的冒險者現在怎麼樣了」，忍不住打開視窗看一眼
- 玩家會因為一位常駐冒險者的死亡而真正感到心痛
- 玩家會根據想走的陣營路線來選擇性地審核委託
- 閒置冒險者會在公會內走動、產生對話，讓桌面視窗感覺是「活的」
- 閒置冒險者可能自行接取委託板上的任務（不需要玩家推薦），強化 NPC 自主性——但也意味著玩家精心挑選的任務可能被「搶走」

### 核心玩法（要建立的系統）

1. **委託系統** — 委託出現、審核、推薦、結算的完整流程
2. **冒險者系統** — 冒險者有個性（Trait）、能力值、接受/拒絕機率、死亡判定
3. **間接控制三槓桿** — 公會建築升級、公會職員屬性加成、推薦成功率
4. **經濟系統** — 傭金收入、建設支出、賠償損失、公會倒閉風險
5. **陣營劇情系統** — 委託選擇 + 建設方向 + 職員組合 → 觸發劇本分支

## 玩家動機

### 主要心理需求滿足

| 需求 | 遊戲如何滿足 | 重要性 |
| ----------------------------------------- | -------------------------------------------------------------- | ---------- |
| **自主性** | 玩家自由決定審核哪些委託、建設什麼、招誰。每個選擇塑造公會方向 | 核心 |
| **勝任感** | 學會評估風險、最佳化資源配置、理解 NPC 行為模式 | 輔助 |
| **歸屬感** | 與冒險者的情感連結——他們不是數字，是有故事的人 | 核心 |

### 玩家類型偏好

- [x] **成就者** — 公會從破舊到傳奇的成長、冒險者收集、聲望累積
- [x] **探險者** — 發現冒險者的故事、理解間接控制的系統深度
- [ ] **社交者** — 單人遊戲，但與 NPC 的情感連結提供部分滿足
- [ ] **征服者** — 非競爭型遊戲

### 可能進入Flow的原因

- **入門曲線**：第一個委託是教學任務，低風險，讓玩家理解推薦→等待→結算的流程
- **難度調度**：隨公會聲望提升，高難度委託出現，但也帶來更高報酬與劇情推進
- **反饋清晰度**：結算時清楚顯示成敗原因（冒險者能力 vs 任務需求 vs 運氣）
- **失敗恢復**：冒險者死亡是永久的，但新冒險者會持續出現。失敗是學習風險評估的代價

## 核心循環（Core Loop）

### 只看一眼（瞥見循環 — 30 秒）

打開桌面視窗 → 看到冒險者在公會走動 → 查看通知（新委託到來 / 任務結算）→ 快速掃一眼狀態。這個「偷看一眼」的動作本身就是滿足的——你的公會是活的。

### 短期（決策循環 — 3-5 分鐘）

審核新委託（決定接或不接）→ 推薦任務給合適的冒險者（考慮能力、Trait、接受率）→ 處理結算通知（收傭金 / 面對傷亡）→ 考慮是否招募新人或升級設施。「再看一單」的心理在這裡運作。

### 遊戲時段（管理時段 — 10-15 分鐘）

規劃公會發展方向 → 調整職員配置 → 審視冒險者整體狀態 → 處理累積事件 → 做出影響陣營走向的關鍵決策。自然停止點：所有委託都已處理、所有結算都已確認。

### 長期進度

公會從破舊小屋 → 知名冒險者據點 → 傳奇殿堂。陣營劇情隨經營決策推進。冒險者來來去去——有的成為傳奇，有的再也沒回來，他們的名字留在公會的歷史中。

### 留存標的

- **好奇心**：派出去的冒險者結果如何？陣營劇情下一步會怎麼發展？
- **投入感**：你培養了半個月的精英冒險者，你不想看到他死
- **情感連結**：與 NPC 冒險者的羈絆（非社交功能，但情感等效）
- **精通感**：學會更精準地評估風險，最佳化推薦策略

## 遊戲支柱

### 支柱 1：間接控制（Indirect Control）

玩家透過公會建設、職員招募、委託審核來影響結果，永遠不直接指揮冒險者的行動。

_設計測試_：「這個功能讓玩家直接控制 NPC 行為嗎？」→ 如果是，重新設計為間接影響。

### 支柱 2：命運由你塑造（Fate Shaped by You）

故事沿設計好的陣營劇本發展，但玩家的委託審核、建設選擇、職員招募決定了走向哪條路線。

_設計測試_：「這個劇情分支是否由玩家的經營決策觸發？」→ 如果不是，改為經營驅動。

### 支柱 3：有重量的決策（Decisions with Weight）

每個選擇都有真實後果——冒險者會死、金幣會虧、公會可能倒閉。好的決策降低風險但不消除風險。

_設計測試_：「這個決策有沒有可能導致不可逆的損失？」→ 如果完全無風險，加入代價。

### 支柱 4：桌面原生（Desktop Native）

遊戲融入工作環境，3-5 分鐘完成一輪決策，透明視窗前置於桌面。不打斷工作，而是成為工作日常的一部分。

_設計測試_：「這個操作能在 3 分鐘內完成嗎？」→ 如果不能，拆分或簡化。

### 不追求的內容

- **不直接戰鬥操控**：沒有玩家手動操作的戰鬥畫面。這會損害「間接控制」和「桌面原生」。
- **不無限容錯**：沒有免費復活、自動回血、無風險任務。這會損害「有重量的決策」。
- **不開放沙盒敘事**：故事不是隨機產生，而是沿陣營劇本推進。這會損害「命運由你塑造」的設計一致性。
- **不長時間專注操作**：沒有需要 10 分鐘以上不間斷操作的功能。這會損害「桌面原生」。

## 風險與未解決的問題

### 設計風險

- 「推薦而非指派」可能讓玩家覺得缺乏控制感——需要讓玩家清楚感受到建設/職員對推薦成功率的影響
- 3-5 分鐘的 session 可能不足以產生有意義的決策深度——需要精心設計資訊呈現的效率

### 技術風險

- 桌面透明視窗在 Unity 中需要原生插件或 Windows API 調用（可延後至 Post-Jam）
- 離線時間的任務結算需要可靠的時間差計算系統

### 市場風險

- 掛機遊戲市場競爭激烈，需要靠「情感連結 + 日系美術」做出差異化
- 桌面前置視窗遊戲是小眾品類，受眾規模不確定

### 規模風險

- 單人一個月的冒險者角色設計量可能不足以支撐敘事深度
- 陣營劇情系統可能在 Game Jam 期間做不完——準備好降級為線性劇情

### 未解決的問題

- NPC 接受/拒絕的機率模型如何設計才能讓玩家覺得「公平但有個性」？→ 需要數值原型驗證
- 冒險者死亡的頻率如何調整才能「心痛但不絕望」？→ 需要 playtest 數據
- 桌面透明模式的 UX 如何處理（點擊穿透、最小化、通知）？→ 需要技術原型
- 閒置冒險者自主接單的機率與頻率如何設計？接到不適合的任務時玩家能否阻止？→ 需要搭配 NPC Logic 系統設計
- 公會職員的數值加成（推薦接受率 +5%、傭金率 +2%）是否足以讓玩家感受到差異？→ 需要 playtest 驗證
- 陣營路線的 styleTag 累積門檻應設為多少？→ 待 `/design-system faction-story` 定義

## MVP 定義（MVP Definition）

**核心假設**：「透過間接控制（推薦而非指派）管理有個性的冒險者，配上有真實後果的任務結算，在 3-5 分鐘的桌面掛機情境下是有趣的。」

**MVP 所需內容**：

1. 委託系統 — 委託生成、審核、推薦給冒險者
2. 冒險者系統 — 含 Trait、能力值、接受/拒絕機率、死亡判定
3. 即時計時 + 結算系統 — 派遣後等待、自動結算、顯示結果
4. 基礎經濟 — 傭金收入、任務支出、公會資金管理
5. 桌面透明視窗 — 核心賣點，支柱 4 的技術基礎

**明確不包含在 MVP 中**（延後至後續）：

- 公會建築升級系統（影響力槓桿之一，但 MVP 可先驗證核心循環）
- 公會職員系統
- 陣營劇情分支
- 音效與音樂

### 規模分級（Scope Tiers）

| 分級                 | 內容                               | 功能                 | 時程         |
| ------------------ | -------------------------------- | ------------------ | ---------- |
| **MVP**            | 5 種委託、4 位冒險者                     | 委託 + 冒險者 + 結算 + 經濟 | 開發 Day 1-3 |
| **垂直切片**           | 10 種委託、6 位冒險者、3 座建築              | + 公會建設 + 職員 + 招募   | 開發 Day 4-6 |
| **Game Jam 完成版**   | 15-20 種委託、8-10 位冒險者、5 座建築、1 條劇情線 | + 陣營劇情 + 桌面模式      | 開發 Day 7-8 |
| **美術置入**           | 角色立繪、UI 素材、背景                    | 素材替換、視覺調整          | Day 9-13   |
| **Debug**          | —                                | 修復、平衡調整、最終測試       | Day 14-15  |
| **完整願景**（Post-Jam） | 50+ 種委託、20+ 位冒險者、完整建設樹、雙陣營劇情     | + 培養 + 隊伍 + 城鎮     | 3-6 個月     |

| 維度 | 範圍 | 說明 |
| ---- | ---- | ---- |

### 冒險者系統（Adventurer System）

**招募機制**：
- **新手自薦**（F/E 階）：自動刷新候選池（4 人/批），免費接納。刷新間隔依招募板等級（初始 24h → 最高 6h）
- **老手邀請**（D~S 階）：花費金幣邀請，加權分布（D:40% / C:30% / B:18% / A:9% / S:3%）
- 名冊上限：依公會等級 4→6→9→12→15 人

**老手邀請費用與條件**（[已棄用，以 【FT-01】adventurer-recruitment.md 為當前權威]）：

| ---- | ---- | ---- |

| C | 250g | ≥20 |
| B | 500g | ≥50 |
| A | 1000g | ≥80 |
| S | 2000g | ≥100 |

### 派遣與結算 — 關鍵公式（Dispatch & Resolution — Key Formulas）

**等級差**：`rankDiff = clamp(RANK_INDEX(冒險者) − DIFFICULTY_INDEX(任務), −4, +3)`

**基礎成功率（依 rankDiff）**（[已棄用，以 【FT-02】mission-dispatch.md §3.3 為當前權威]）：

| rankDiff | +3 | +2 | +1 | 0 | −1 | −2 | −3 |
| ---- | ---- | ---- | ---- | ---- | ---- | ---- | ---- |
| 成功率 | 95% | 85% | 70% | 55% | 35% | 20% | 10% |

**基礎死亡率（依任務難度）**（[已棄用，以 【C-01】mission-database.md `MissionDifficultyTable` 為當前權威]）：

| F | E | D | C | B | A | S | SS | SSS |
|---|---|---|---|---|---|---|----|-----|
| 2% | 6% | 10% | 13% | 18% | 40% | 50% | 60% | 70% |

> **SS 級**：基礎 willingness 不足（S→SS rankDiff=-1），需要 Trait 擅長（+20% 成功率）才有冒險者願意接單。
> **SSS 級**：特殊任務，需透過特殊規則觸發，不在常規任務池中出現。詳細機制待 `/design-system mission-database` 定義。

**NPC 接單意願**：
```
willingnessScore = finalSuccessRate − finalDeathRate × DEATH_AVERSION(0.5)
effectiveScore = willingnessScore + jitter(±0.10)
if effectiveScore < ACCEPTANCE_THRESHOLD(0.25) → 拒絕
```

### 公會職員系統（Guild Staff System）— Phase 2 新增

> 間接控制三槓桿之一。職員透過「職員面試系統」獲得，與建築升級系統完全分開。招募後提供被動加成，可指派到特定工作欄位獲得額外效果。

**3 種公會職員**：

| 職員 | 被動加成（招募即生效） | 指派欄位效果 | 招募條件 |
| ---- | ---- | ---- | ---- |
| **櫃台小姐** | 名冊容量 +2 | 指派至「公會櫃臺」：候選池刷新間隔 -2 小時 | 公會 Lv1，免費 |
| **委託官** | 推薦接受率 +5% | 指派至「委託板」：委託審核時顯示成功率預估 | 公會 Lv2，200g |
| **會計** | 傭金率 +2%（20%→22%） | 指派至「預備金保險櫃」：賠償率 -2%（10%→8%） | 公會 Lv2，300g |

**設計原則**：
- 職員透過「職員面試系統」招募，與建築升級系統獨立
- 每種職員最多招募 1 位（Jam 版）
- 被動加成在招募後永久生效，不需要指派
- 指派到欄位提供額外加成，但同一時間只能指派到一個欄位
- 指派欄位需要對應公會建設已解鎖（櫃台小姐→公會櫃臺、委託官→委託板、會計→預備金保險櫃）

> 詳細設計待 `/design-system guild-staff` 補完。
>
> **實作更新（2026-04-24 / 2026-04-26 拆分）**：本節 3 種具名職員的設計已改為 **gacha + effect 聚合**架構——職員面試系統採隨機招募池，職能以 `StaffTable.effectIDs / effectValues`（CSV list；原 `tags`）表達，對外仍暴露同名加成 API（`GetStaffWillingnessBonus` / `GetAccountantCommissionBonus` / `GetAccountantPenaltyBonus`）。**2026-04-26 拆分**：原職員系統拆為 `[FT-08] gacha-system.md`（面試 gacha）與 `[FT-12] staff-system.md`（職員運營），加成 API 由 FT-12 提供、面試 gacha 由 FT-08 主管。本節表格保留為**設計意圖參考**；最新規格以 FT-08 / FT-12 GDD 為準。

**Post-Jam 擴充構想：人事系統（Human Resources System）**

> 2026-04-24 於 FT-08 GDD Section 2 討論時確認。Jam 版 FT-08 聚焦「面試驚喜」+「加成可見」（E1 Sensation + E4 Mastery），以下三項情緒面向延後至 Post-Jam，整合為「人事系統」玩法：

- **收集欲 / 圖鑑完成感（E2 Expression + Discovery）**：職員有陣營、tag、rarity 的組合，玩家可透過圖鑑追蹤蒐集進度；稀有 tag 組合（如「會計 + 委託官雙職能」）成為收集目標，重複面試動機由此而生
- **班表感 / 排班經營（E5 Mastery + Fantasy）**：職員三態（工作中 / 再分配 / 休假）的細部管理升級為主動排班——玩家可設定輪班時間表、預排休假、指定何時切換 slot；職員疲勞值 / 好感度可能影響產出
- **薪水壓力 / 破產倒數（E7 Challenge + Submission）**：職員薪水與戰力、滿意度連動——高階職員薪水高但加成多；低薪職員可能因不滿而離職、罷工；破產前的「薪水是否發得出來」成為核心張力

**三角平衡玩法核心**：排班 ↔ 薪水預算 ↔ 職員隊伍組成——三者任一失衡都會引發連鎖後果（排班過密 → 疲勞 → 加成失效；薪水過低 → 離職潮 → 人力斷層；隊伍組成單一 → 加成不均 → 結算體感單調）。

> Jam 版僅保留此系統的最小骨幹：`StaffTable.salary` 欄位、休假態不支薪、每日 UTC 06:00 發薪管線。進階玩法（主動排班、疲勞 / 好感度、離職 / 罷工、圖鑑）全數延後。

## 資料驅動架構（Data-Driven Architecture）

本專案採用完全資料驅動設計：程式碼為通用引擎，所有遊戲內容由 CSV 表格控制。

### 設計原則

- **零硬編碼**：所有數值（公式常數、報酬、成功率等）來自 CSV 表格
- **ID 互參**：系統間透過 ID（primary key / foreign key）互相引用，無字串耦合
- **修改即生效**：新增冒險者/任務/Trait = 新增 CSV row，不需改程式碼
- **隨機化支援**：群組表定義「從哪些選項中抽幾個」，由 DataManager 統一處理

### 資料流

```
Google Sheets（Drive 資料夾）
    ↓ 匯出
CSV 檔案（Assets/Resources/Data/Tables/）
    ↓ Runtime 載入
DataManager（TextAsset → Dictionary<string, T>）
    ↓ ID 查詢
各系統讀取資料 → 執行規則
```

### 表格分類

| 類別 | 表格數 | 用途 |
| ---- | ---- | ---- |
| 系統常數 | 1 | 所有公式關鍵值（key-value） |
| 參數表 | ~20 | 各系統的實體定義（職業、種族、任務、建築等） |
| 文字表 | ~6 | 獨立文字內容（名字池、UI 文字、對話等） |

> 完整表格 schema 見 `design/gdd/systems-index.md`。
